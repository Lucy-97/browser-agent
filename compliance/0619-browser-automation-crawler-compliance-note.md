# 浏览器自动化与网页采集合规备忘录

## Changelog

- 2026-06-19：新增浏览器自动化、网页采集和 PDF 下载相关合规分析，覆盖刑法计算机犯罪边界、数据/个人信息/版权/反不正当竞争风险、产品红线和工程控制建议。

## 一、定位与免责声明

本文是 Browser Agent Local Automation Worker 的产品合规备忘录，不构成正式法律意见。商业上线前，应结合具体数据源、访问规模、客户授权、下载内容、数据用途、部署区域和合同条款，由中国法执业律师出具审查意见。

当前 Worker 的核心形态是：

1. 用户本机运行 Worker。
2. Worker 控制本机 Chromium/Chrome 模拟用户访问网页。
3. 平台下发任务，Worker 主动 pull，平台不反向连接用户机器。
4. 用户账号密码默认只进入本机浏览器或本地凭证，不上传平台。
5. Worker 上传 HTML、截图、metadata、PDF 等 artifact 到平台。

## 二、初步结论

“用浏览器模拟普通用户访问公开网页或用户有权访问的网页”本身，通常不能直接等同于“破坏计算机信息系统罪”。破坏计算机信息系统罪更关注删除、修改、增加、干扰系统功能或数据，并造成相应后果。

但“模拟人的行为”不是免责理由。爬取、下载、自动化操作仍可能触发刑法、网络安全、数据安全、个人信息保护、著作权、反不正当竞争、合同违约和平台规则风险。风险取决于授权范围、数据性质、访问方式、访问频率、是否绕过技术措施、是否造成损害、是否商业化使用以及是否涉及个人信息或非公开数据。

## 三、主要法律风险地图

### 1. 计算机信息系统犯罪风险

重点关注《中华人民共和国刑法》第 285 条、第 286 条相关风险：

1. 非法侵入计算机信息系统。
2. 非法获取计算机信息系统数据。
3. 非法控制计算机信息系统。
4. 破坏计算机信息系统功能、数据或应用程序。

低风险方向：

1. 仅访问公开页面或用户已授权、已登录后可正常浏览的页面。
2. 不绕过登录、权限、验证码、付费墙、加密、签名、风控和反爬技术措施。
3. 不注入脚本破坏对方页面，不修改对方系统数据。
4. 严格限速，不造成服务异常、资源耗尽或可感知损害。

高风险方向：

1. 绕过验证码、风控、访问控制或反爬机制。
2. 伪造身份、盗用 token、共享账号池、绕过付费权限。
3. 大规模并发访问导致对方系统变慢、宕机或成本显著增加。
4. 获取非公开数据、内部接口数据、他人账号数据或受保护数据库。

### 2. 个人信息与数据安全风险

如果采集内容包含自然人姓名、邮箱、电话、账号标识、位置、头像、履历、论文作者联系方式等，可能构成个人信息处理，需要关注《个人信息保护法》的合法基础、最小必要、目的限定、告知、同意、删除、访问、更正和跨境提供等要求。

低风险方向：

1. 优先采集论文标题、DOI、作者名、期刊、年份、摘要、引用链接等科研元数据。
2. 只采集任务目的必要字段。
3. 对个人信息字段做分类、脱敏、访问控制和留存期限管理。
4. 对外提供数据前做合规评估和去标识化/匿名化处理。

高风险方向：

1. 批量采集个人联系方式、账号主页、社交关系、行为轨迹。
2. 将公开个人信息用于与原公开目的无关的营销、画像、自动化决策。
3. 把境内收集的个人信息未经评估提供给境外主体。

### 3. 著作权与下载内容风险

论文 PDF、出版社网页、数据库内容和图表可能受著作权、数据库合同或平台许可限制。即使 Worker 能下载，也不代表平台有权复制、存储、分发或训练模型使用。

建议边界：

1. PDF 下载仅服务用户授权的科研解析任务。
2. 默认不对外公开 PDF 原文，不做二次分发。
3. 保留来源 URL、下载时间、用户任务、授权上下文和 artifact 审计。
4. 对明显受付费墙保护或禁止下载的内容，默认不绕过。
5. 后续如接入机构订阅数据库，应由客户确认授权范围和许可条款。

### 4. 反不正当竞争与合同风险

即使未达到刑事风险，批量爬取商业平台、学术数据库、内容平台或竞争对手数据，仍可能引发反不正当竞争、合同违约或侵权争议。

高风险特征：

1. 绕过 robots、服务条款、API 限制或反爬策略。
2. 高并发、大规模、持续性采集。
3. 采集对方投入建设的商业数据库并替代其服务。
4. 对对方服务器造成负担或对正常经营造成影响。
5. 商业化输出与原平台形成替代或竞争关系。

## 四、产品红线

Browser Agent Worker 应默认禁止或人工升级审批以下任务：

1. 绕过验证码、风控、付费墙、登录权限或技术保护措施。
2. 使用账号池、代理池、盗用 token、撞库或批量注册账号。
3. 未经授权采集非公开数据、账号数据、后台数据或个人敏感信息。
4. 高频、大规模、分布式访问目标站点。
5. 自动发布、付款、授权、删除、提现、转账等敏感动作无人工确认。
6. 下载或传播明显受版权限制且无授权的完整内容。
7. 面向第三方提供“规避反爬”“验证码绕过”“批量抓取付费内容”等能力。

## 五、工程控制建议

### 1. Policy Gate

每个 job 必须带 policy：

```json
{
  "allowed_domains": ["scholar.google.com", "arxiv.org"],
  "allowed_actions": ["goto", "click", "fill", "extract", "download", "screenshot"],
  "max_requests_per_minute": 6,
  "max_downloads": 3,
  "requires_human_confirmation_before_sensitive_action": true
}
```

Worker 执行前必须校验：

1. 目标域名是否在白名单。
2. 动作是否在白名单。
3. 下载 MIME/type 是否允许。
4. 是否超过速率、时间、下载数和重试上限。
5. 是否触发人工确认。

### 2. Human-in-the-loop

以下场景必须暂停并创建 `manual_action`：

1. 验证码、登录、二次验证。
2. 站点提示异常流量、禁止自动访问或访问受限。
3. 需要发布、付款、授权、删除、上传外部内容。
4. 页面无法确定动作风险。

### 3. 审计与可追溯

每个 run 至少保留：

1. job payload、policy、用户、设备、时间。
2. 每步 checkpoint。
3. 页面截图、HTML snapshot、下载 URL、artifact sha256。
4. manual action 创建和 resolve 记录。
5. 失败原因、重试次数和限速记录。

### 4. 数据治理

平台侧应实现：

1. artifact 分类：PDF、HTML、截图、metadata、trace、error_context。
2. 敏感字段识别和脱敏。
3. 按租户/用户/任务做权限隔离。
4. artifact 留存期限和删除机制。
5. 对 PDF 原文、截图等高风险 artifact 做下载鉴权。

## 六、当前项目状态

已完成：

1. Worker 使用本机浏览器，不把第三方账号密码上传平台。
2. Worker 主动 pull job，平台不反向连接用户机器。
3. job policy 已有 `allowed_domains` 基础校验。
4. 目标站点遇到验证码、异常流量或登录失效时会进入 `manual_action`。
5. Run、artifact、manual action 已有平台记录。
6. PDF artifact 已记录 metadata、sha256、size 和本地存储路径。

仍需补齐：

1. 更细的 `allowed_actions`、速率、下载类型、下载数量 policy enforcement。
2. Robots/站点条款提示和数据源授权记录。
3. 敏感数据识别、脱敏、留存和删除策略。
4. 面向用户的授权确认和用途声明。
5. 高风险动作审批流。
6. 合规审计报表和导出能力。
