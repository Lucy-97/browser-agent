# Browser Agent 商业场景扩展与架构解耦需求文档 (BRD)

## 文档信息
- **日期**: 2026-06-27
- **状态**: 草案 / 规划中
- **相关方**: 产研团队、业务运营团队

## 1. 背景与业务契机

前期在“AI 4 Science”领域（文献自动抓取、专业知识三元组抽取）的实践，充分验证了利用 Agent 模拟人类真实浏览器行为（非传统技术类爬虫）在复杂交互网页中获取数据和执行动作的高度可行性。

基于这一能力底座，我们发现其商业价值远不仅限于学术文献下载，理论上**“只要人类能通过浏览器看到和操作的内容，Agent 都能看到并模拟操作”**。由此引申出极具商业化潜力的两大扩展方向：

1. **泛内容自动化运营**：例如在 Reddit 刷帖、Twitter 账号自动化监控与互动。
2. **版权侵权检索与取证（如微短剧）**：针对全网盗版泛滥的问题，通过部署多台 Agent 机器进行高频次全网巡查，帮助版权方（如微短剧出品方）定位盗用视频并截图取证，最终形成对 B 端的“侵权检测付费服务”。

## 2. 核心架构解耦策略

随着商业场景的增多，如果将各行业的定制化逻辑（社交媒体发帖规则、短剧盗版识别逻辑等）全部塞入基础引擎中，会导致代码库迅速膨胀且极难维护。因此，必须在架构和代码管理上执行严格的**解耦策略**。

### 2.1 独立分支开发（业务隔离）
- **策略**：将泛自动化运营（如版权检索系统）与 AI 4 Science（文献下载）的具体业务需求在分支层面彻底隔离。
- **执行**：在同一个代码库下，通过独立分支进行开发。例如 `feature/browser-agent` 分支专门用于开发自动化运营与版权检索，而 `feature/qiyuan` 分支则专注于文献抽取功能。功能在各自分支独立演进，避免业务逻辑相互耦合。

### 2.2 组件共享与技术栈一致性（底座层）
- **策略**：无论是 `feature/browser-agent` 还是 `feature/qiyuan`，都基于同一个底层基础架构。
- **共享机制**：底层的核心组件（如基于 Playwright 的人类行为模拟框架、反爬虫绕过策略、分布式 Worker 调度、通用 API 等）在分支间保持共享与同步，确保技术栈的高度一致性。
- **收益**：避免重复“造轮子”；底层一旦攻克某种新型反爬机制或完成性能优化，可通过合并/拣选（Cherry-pick）让各个业务分支直接受益。

## 3. 重点业务场景定义

### 3.1 场景一：短剧版权侵权检索（首个商业化 PoC 推荐）
- **用户痛点**：微短剧上架后常在不知名的小网站或长视频平台上被盗用，人工巡查成本极高且容易漏网。
- **Agent 解决方案**：
  - 业务层：生成“剧名 + 免费全集”、“剧名 + 百度网盘”等搜索指令下发给底座。
  - 底座层：Agent 控制浏览器在全网主流平台和搜索引擎执行检索。
  - 取证动作：发现疑似盗用链接后，Agent 自动进入页面，截取网页全屏（Screenshot Artifact），保存原始来源 URL，甚至捕获视频 DOM 节点元素。
- **产品形态**：B 端 SaaS 版权监测服务平台。

### 3.2 场景二：海外社交媒体自动化运营
- **用户痛点**：出海企业在 Reddit、Twitter、Discord 等平台进行社群运营需要大量人工账号维护，且 API 限制极严。
- **Agent 解决方案**：
  - 结合大模型的内容生成能力，Agent 模拟真实浏览器环境（包含特定的 Canvas/指纹伪装），自动进行点赞、评论、刷帖和舆情监控。
  - 处理突发情况：遇到强阻断验证码时，触发 `manual_action` 将验证码截图推送给人工后台，人工点击通过后 Agent 继续执行（人机协同）。

## 4. 下一步行动计划 (Next Steps)

1. **巩固底座**：确保当前的 `feature/browser-agent` 分支具备良好的 API 封装与对接能力，以便外部独立服务可以轻松下发任务并获取 Artifact（截图、数据）。
2. **规划独立分支**：明确 `feature/browser-agent` 作为自动化运营与版权检索的专属分支，在其中搭建相关的业务模型和任务流。
3. **闭环验证**：在业务侧硬编码几个已知可能存在侵权的检索词，驱动底座跑通一次“检索 -> 发现目标页面 -> 截图取证 -> 返回报告”的完整链路。
