## 1. 背景与业务契机

前期的复杂网页自动化实践验证了：通过 Browser Agent 模拟真实用户在浏览器中的观察和操作，可以覆盖传统接口或规则型爬虫难以处理的动态页面、登录态流程和人机协作场景。

这套能力的商业价值不仅是“读取网页”，更在于完成可审计的网页任务闭环。当前优先探索两个方向：

1. **版权侵权检索与取证**：面向短剧等内容版权方，自动发现疑似盗用页面，保存来源 URL、截图、页面信息和执行轨迹，形成可复核的线索报告。
2. **泛内容自动化运营**：面向 Reddit、X、YouTube、TikTok、Instagram 等平台，在明确授权与平台规则边界内执行内容发布、账号维护和运营辅助任务。

## 2. 核心架构解耦策略

业务场景增加后，不能把平台规则、版权判断或社媒发布流程直接塞进浏览器运行时。仓库统一采用单一主线和模块化扩展机制。

### 2.1 单一主线

- `main` 是唯一长期开发主线，保存可运行、可验证的 Browser Agent 平台。
- 日常功能可使用短期分支开发，验收后合入 `main` 并删除，不维护平行产品分支。
- README、部署脚本和交接文档以 `main` 的现状为准，避免出现不同分支对应不同产品或环境的假设。

### 2.2 模块隔离

- **底座层**：设备绑定、任务调度、Playwright 浏览器运行时、checkpoint、artifact、manual action、trace 和通用 API。
- **扩展层**：使用 Worker extension 和 adapter 注册具体能力，例如 `generic.browser_agent`、社媒上传和微信资料同步。
- **策略层**：通过 policy template、allowed domains、人工确认和审计字段约束具体任务，不把平台特例写入通用执行器。
- **收益**：底座只维护稳定的执行协议；具体业务可以独立测试、按需启用，并在失败时提供清晰的错误与人工接管边界。

## 3. 重点业务场景定义

### 3.1 场景一：短剧版权侵权检索（首个商业化 PoC）

- **用户痛点**：短剧上线后可能在小型网站或视频平台被盗用，人工巡检成本高且容易漏网。
- **任务输入**：作品名、别名、版权方、目标站点或允许域名、检索关键词和时间范围。
- **执行过程**：Browser Agent 检索目标页面，记录 URL、时间、页面标题、匹配线索和关键 DOM 信息。
- **取证输出**：页面截图、执行 trace、结构化线索和可下载报告；所有结果必须允许人工复核，不能把自动匹配直接表述为法律结论。
- **产品形态**：面向版权方的 B 端监测与线索管理服务。

### 3.2 场景二：海外社交媒体自动化运营

- **用户痛点**：多平台内容发布和账号维护需要大量重复操作，且各平台 API 能力与限制不一致。
- **Agent 方案**：复用真实浏览器 profile，按平台 adapter 执行导航、上传、填写元数据、可见性设置和提交。
- **安全边界**：默认不绕过验证码或平台风控；遇到登录失效、验证码、敏感发布动作时创建 `manual_action`，由人工确认后继续。
- **审计要求**：记录任务发起人、目标账号、内容、可见性、关键步骤、最终结果和 artifact，不保存第三方账号密码、Cookie 或 Session 明文。

## 4. 下一步行动计划

1. **完成版权检索业务建模**：定义作品、关键词、目标站点、疑似侵权线索和取证 artifact 的最小数据模型与 API。
2. **跑通首个真实 PoC**：从 Web 创建版权检索任务，完成“检索 → 发现候选页面 → 截图取证 → Admin 复核 → 导出结果”的闭环。
3. **补齐安全与合规约束**：为 allowed domains、访问频率、人工确认、数据留存和删除建立默认 policy，并记录审计日志。
4. **稳定现有 adapter**：为社媒上传和微信资料同步补 fixture、错误码、登录状态识别、重试边界和人工接管验证。

当前底座、Windows 本地运行和 Browser Agent 确定性端到端验收已经完成；下一阶段不再规划新的长期业务分支，工作重点转向版权检索 PoC 和生产化边界。
